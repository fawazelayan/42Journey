# C++ Code Formatting & Style Guidelines

This document outlines the specific code formatting, indentation, and structure rules to be followed in this repository.

---

## 1. Indentation and Alignment
* **Use Tabs Only**: Indent and align code strictly using tabs (`\t`).
* **No Mixed Spaces**: Do not combine tabs and spaces for alignment. For continuation lines, use multiple tabs (e.g., `\t\t`).
  
```cpp
// Correct (contributions use double tabs)
AForm::AForm()
	: name("Default Form"),
		isSigned(false),
		signGrade(150),
		execGrade(150) {}
```

---

## 2. Braces and Brackets `{}` Alignment
* **Allman Style**: For blocks (`if`, `else`, `try`, `catch`, loops, functions), place both the opening `{` and closing `}` on separate lines, vertically aligned with the block keyword.
* **Inline Construct Destruct Exception**: Empty constructor/destructor bodies can be written inline as ` {}` at the end of the initialization list.

```cpp
// Standard block alignment
void testNormalBureaucrat()
{
	try
	{
		// ...
	}
	catch (const std::exception& e)
	{
		// ...
	}
}
```

---

## 3. Single-Command Blocks
* **Conditional and Loop Blocks**: If an `if`, `else`, `for`, or `while` block contains exactly **one statement**, you **must omit** the braces `{}`.
  * **Exception for Nested / Complex Blocks**: Even if a block technically contains only a single statement (such as a nested `if` inside a `for` loop, or nested `if`s/loops), you **must use** braces `{}` for the outer block if the structure spans multiple lines or contains other blocks. This ensures visual clarity for the viewer.
* **Try/Catch Blocks**: In C++, `try` and `catch` compound statements strictly require `{}`. These braces **must not** be omitted.

```cpp
// Correct: use braces because the nested if statement spans multiple lines/contains a block
for (int i = 0; i < MAX_VAL; i++)
{
	if (mirror[i] != numbers[i])
	{
		std::cerr << "didn't save the same value!!" << std::endl;
		return 1;
	}
}

// Correct: braces omitted for simple, single-statement if-else on single lines
if (rand() % 2 == 0)
	std::cout << this -> target << " success" << std::endl;
else
	std::cout << this -> target << " failure" << std::endl;

// Correct: braces omitted for simple, single-statement loops
for (int i = 0; i < 3; ++i)
	delete forms[i];
```

---

## 4. Return Statements
* Do not wrap the returned expression in parentheses.
  
```cpp
// Correct
return this -> name;
return *this;

// Incorrect
return (this -> name);
return (*this);
```

---

## 5. Stream Output (`std::cout`, `std::cerr`, `os`)
* **Simple Outputs**: Keep simple print statements and their `std::endl` on a single line.
* **Multi-Field Outputs**: For printing structured multi-field objects, place each field insertion on a separate line, indented with tabs.

```cpp
// Simple print (kept on single line)
std::cout << "Created: " << b1 << std::endl;

// Multi-field print (split on lines using tabs)
std::ostream& operator<<(std::ostream& os, const AForm& form)
{
	os << "Form: " << form.getName()
		<< ", signed: " << (form.getIsSigned() ? "yes" : "no")
		<< ", sign grade required: " << form.getSignGrade()
		<< ", execute grade required: " << form.getExecGrade();
	return os;
}
```

---

## 6. Constructor Initialization Lists
* If there is more than **one member initializer**, split each initializer onto a separate line.

```cpp
Bureaucrat::Bureaucrat(const std::string& name, int grade)
	: name(name),
		grade(grade)
{
}
```

---

## 7. Performance Include Rules
* **Header Files (`.hpp`)**: Use the lightweight `<iosfwd>` header instead of full stream headers like `<ostream>` or `<iostream>` to optimize compile times when only stream references are declared.
* **Source Files (`.cpp`)**: Include the full stream headers (`<ostream>`, `<iostream>`) inside source files where stream insertions are executed.
* **Minimal Includes**: Avoid redundant header inclusions. Only include headers that are strictly necessary for the code inside the file to compile and function.

---

## 8. Documentation Comments
* **Headers**: Organize header files using brief section comments:
  - `// Constructors & Destructor`
  - `// Getters`
  - `// Grade Modifiers`
  - `// Form Operations`
  - `// Exception Classes`
* **Mains**: Add short, simple explanatory comments to describe test functions.

---

## 9. Naming Conventions and Test Outputs
* **Naming Conventions**: Variables and function names must be written in lowerCamelCase. Do not add prefix underscores `_` to variables (e.g., member variables) unless strictly necessary to resolve a naming conflict.
* **Test Output Demarcation**: In test scripts or main executable outputs, visually separate the expected outputs and the actual outputs with a distinct separator line (e.g., `----------------------------------`) and heading labels for clarity.

```cpp
// Correct: lowerCamelCase, no leading underscores
int shortestSpan = 0;
std::vector<int> sortedElements;

// Correct: Visual separation of expected vs actual test outputs
std::cout << "Expected Output:" << std::endl;
std::cout << "  Shortest = 2, Longest = 14" << std::endl;
std::cout << "----------------------------------" << std::endl;
std::cout << "Actual Output:" << std::endl;
std::cout << "  Shortest span: " << sp.shortestSpan() << std::endl;
```

---

## 10. Additional formatting details
* **Access Specifiers Indentation**: Indent `private:` and `public:` using one tab, and indent everything under them with another tab (two tabs total).
* **Empty Class and Function Definitions**: For empty classes and empty member functions (e.g. empty constructors or destructors), place the braces inline on the same line (e.g. `class A : public Base {};` or `A::A() {}`).
* **Loops and Conditions containing complex blocks**: Always include braces `{}` for any loops or conditional statements containing `if` statements, other loops, or complex/long nested structures (even if they constitute a single statement technically).
* **Space around 'this' arrow**: Always separate the 'this' pointer and its member access arrow operator by exactly one space before and one space after the arrow (e.g., `this -> member` instead of `this->member`).
* **Header Guards Formatting**: Header guards must contain a single space between the '#' and 'define' keyword (e.g., `#ifndef WHATEVER_HPP` followed by `# define WHATEVER_HPP`).
* **Include Directives Formatting**: Do not place any space between the '#' and 'include' keyword (e.g., use `#include <exception>` instead of `# include <exception>`).
* **Template Implementation File Guards**: Template implementation files (`.tpp`) must not use standard double-inclusion guards (e.g., `#ifndef WHATEVER_TPP`). Instead, they must enforce being included only by their header file using an `#error` guard directive checking if the parent `.hpp` guard is defined (e.g., `#ifndef WHATEVER_HPP \n # error "..." \n #endif`).
* **No Trailing Empty Lines**: Ensure there is no new empty line or newline character at the very end of any source or header file. The file must terminate exactly on the last character of the code (e.g., `}` or `#endif`), leaving no empty/blank lines in the editor. Do a post-write check/script to strip any trailing newlines if necessary.
